# Remote sudo with Touch ID via vt auth

Use `vt auth` to trigger Touch ID on your macOS machine when running `sudo` on a remote Linux server. If macOS is unreachable or Touch ID is rejected, sudo falls back to normal password authentication.

## How it works

```
macOS                                    Remote Linux Server
┌─────────────────────┐                  ┌──────────────────────────┐
│ vt SSH agent        │  SSH agent fwd   │ sudo somecommand         │
│ (~/.ssh/vt.sock)    │◄────────────────│   └─ PAM                 │
│                     │                  │       └─ pam_exec        │
│ receives auth@vt    │                  │           └─ vt auth     │
│ triggers Touch ID   │                  │               └─ SSH_AUTH│
│ returns ok/fail     │────────────────►│                  _SOCK   │
└─────────────────────┘                  └──────────────────────────┘
```

1. You SSH from macOS to the remote server with agent forwarding (`ssh -A`)
2. On the server, `sudo` triggers PAM, which runs `vt auth` via `pam_exec`
3. `vt auth` sends an `auth@vt` extension through `SSH_AUTH_SOCK` (forwarded back to macOS)
4. The vt SSH agent on macOS receives the extension and prompts Touch ID
5. If approved, `vt auth` exits 0 and sudo proceeds
6. If rejected/unreachable, `vt auth` exits non-zero, PAM falls through to password prompt

## Prerequisites

- macOS machine running `vt serve` (or standalone `vt ssh agent`) with `SSH_AUTH_SOCK=~/.ssh/vt.sock`
- Remote Linux server with `vt` binary installed
- SSH agent forwarding enabled between macOS and the server
- `VT_AUTH` token (generated by `vt init`)

## Step 1: macOS setup

Ensure the vt SSH agent is running and set as your SSH agent:

```bash
# Add to your shell profile (~/.zshrc or ~/.bashrc)
export SSH_AUTH_SOCK=~/.ssh/vt.sock
```

Start the agent (if not using `vt serve`):

```bash
vt ssh agent
```

Or via `vt serve` (which starts the SSH agent automatically):

```bash
vt serve
```

## Step 2: Enable SSH agent forwarding

Edit `~/.ssh/config` on your macOS machine:

```
Host your-server
    HostName your-server-ip-or-hostname
    ForwardAgent yes
```

Or connect with the `-A` flag:

```bash
ssh -A user@your-server
```

Verify agent forwarding works on the remote server:

```bash
ssh-add -l   # Should list your vt-managed keys
```

## Step 3: Install vt on the remote server

Cross-compile for Linux on macOS:

```bash
# For x86_64 Linux
cargo build --release --target x86_64-unknown-linux-gnu
scp target/x86_64-unknown-linux-gnu/release/vt user@server:/usr/local/bin/vt

# For aarch64 Linux
cargo build --release --target aarch64-unknown-linux-gnu
scp target/aarch64-unknown-linux-gnu/release/vt user@server:/usr/local/bin/vt
```

Or download from [GitHub Releases](https://github.com/timqi/vt/releases).

Make sure it's executable:

```bash
sudo chmod +x /usr/local/bin/vt
```

## Step 4: Preserve SSH_AUTH_SOCK through sudo

By default, `sudo` strips most environment variables including `SSH_AUTH_SOCK`. You must configure sudoers to preserve it.

Run `sudo visudo` and add:

```
Defaults env_keep += "SSH_AUTH_SOCK"
```

Without this, the forwarded agent socket is invisible to `vt auth` running inside `pam_exec`, and it will always fail (falling back to password).

## Step 5: Create the PAM helper script

Create `/usr/local/bin/vt-sudo-auth.sh`:

```bash
#!/bin/bash

export VT_AUTH="your-base64-token-here"

# SSH_AUTH_SOCK must be available (forwarded agent)
if [ -z "$SSH_AUTH_SOCK" ]; then
    exit 1  # No forwarded agent, fall through to password
fi

# Timeout prevents hanging if macOS is asleep or unreachable.
# PAM_USER and PAM_SERVICE are set by pam_exec automatically.
timeout 30 /usr/local/bin/vt auth \
    --reason "sudo ${PAM_SERVICE:-sudo} by ${PAM_USER:-unknown}" 2>/dev/null
```

Replace `your-base64-token-here` with the actual `VT_AUTH` value from `vt init`.

**Security warning**: This token also authorizes `encrypt@vt` and `decrypt@vt` operations on the SSH agent. Anyone who can read this file can decrypt all vt-protected secrets. Keep the script strictly root-only (chmod 700).

Set permissions:

```bash
sudo chmod 700 /usr/local/bin/vt-sudo-auth.sh
sudo chown root:root /usr/local/bin/vt-sudo-auth.sh
```

## Step 6: Configure PAM

Edit `/etc/pam.d/sudo` and add the `pam_exec` line **before** the existing auth lines:

```
# Touch ID via vt (falls through to password if unavailable)
auth    sufficient    pam_exec.so quiet /usr/local/bin/vt-sudo-auth.sh
# Original auth (password fallback)
@include common-auth
```

The `sufficient` keyword means:
- If `vt auth` succeeds (exit 0): sudo proceeds immediately, skipping password
- If `vt auth` fails (any non-zero exit): PAM continues to the next auth module (password prompt)

### Understanding the fallback behavior

| Scenario | What happens |
|---|---|
| Touch ID approved on macOS | `pam_exec` returns success, sudo proceeds |
| Touch ID rejected on macOS | `pam_exec` returns failure, falls through to password |
| macOS is asleep / lid closed | `timeout 30` expires, falls through to password |
| macOS is unreachable (network) | SSH agent socket errors, falls through to password |
| No agent forwarding (`SSH_AUTH_SOCK` unset) | Script exits 1 immediately, falls through to password |
| Local sudo (not over SSH) | No `SSH_AUTH_SOCK`, falls through to password |
| `VT_AUTH` not set in script | `vt auth` fails, falls through to password |

## Step 7: Test it

1. On macOS, make sure the vt SSH agent is running:
   ```bash
   export SSH_AUTH_SOCK=~/.ssh/vt.sock
   vt serve  # or: vt ssh agent
   ```

2. SSH to the remote server with agent forwarding:
   ```bash
   ssh -A user@your-server
   ```

3. Run sudo:
   ```bash
   sudo whoami
   ```

4. You should see a Touch ID prompt on your macOS machine with a message like:
   ```
   bio auth: sudo sudo by user from your-server
   ```

5. Approve with Touch ID — sudo proceeds without a password.

6. To test the fallback, reject Touch ID or disconnect the SSH agent — you should get a normal password prompt.

## Troubleshooting

### "SSH_AUTH_SOCK not set" error

- Make sure you SSHed with `-A` or `ForwardAgent yes`
- Check that `SSH_AUTH_SOCK` is preserved through sudo: `sudo env | grep SSH_AUTH_SOCK`
- If missing, verify the `env_keep` line in sudoers (Step 4)

### Touch ID prompt doesn't appear

- Verify the vt SSH agent is running on macOS: `SSH_AUTH_SOCK=~/.ssh/vt.sock ssh-add -l`
- Check that agent forwarding works: run `ssh-add -l` on the remote server
- Check the helper script runs correctly: `sudo /usr/local/bin/vt-sudo-auth.sh; echo $?`

### Always falls back to password

- Check that `VT_AUTH` in `/usr/local/bin/vt-sudo-auth.sh` matches the token from `vt init`
- Make sure the `pam_exec` line is **before** `@include common-auth` in `/etc/pam.d/sudo`
- Check permissions: `ls -la /usr/local/bin/vt-sudo-auth.sh` should show `-rwx------ root root`

### sudo hangs for 30 seconds then asks for password

- macOS is unreachable or asleep — the `timeout 30` is protecting you from an indefinite hang
- Reduce the timeout in the script if 30 seconds is too long
- Consider using `timeout 10` for faster fallback

## Security considerations

### VT_AUTH is a full credential

The `VT_AUTH` token in the helper script authorizes not just `auth@vt` but also `encrypt@vt` and `decrypt@vt`. If the script is compromised, an attacker with access to your forwarded SSH agent could decrypt all vt-protected secrets.

Mitigations:
- Keep the helper script as root:root 700
- Limit which users can sudo (and thus trigger the PAM script)

### No auth caching for auth@vt

Unlike `sign` and `decrypt@vt` which support optional auth caching, `auth@vt` always prompts Touch ID. This is intentional: over SSH agent forwarding, all remote sessions share the same local SSH process. If caching were enabled, one Touch ID approval could silently authorize sudo from any remote session sharing that forwarded agent.

### Prompt text is advisory

The Touch ID prompt text (`host`, `reason`) comes from the remote server and cannot be verified. A compromised server could send any text. The vt agent sanitizes these strings (strips control characters, truncates to 100 chars) but cannot guarantee their truthfulness.

### `sufficient` bypasses other PAM modules

Placing `pam_exec` as `sufficient` before `common-auth` means a successful Touch ID approval skips everything else in `common-auth` — including password checks, account lockout (faillock/pam_tally2), and any second factor. This is the intended behavior (biometric replaces password), but be aware of this if you have other PAM modules in your auth stack.

### macOS password fallback

vt uses the `DeviceOwnerAuthentication` policy, which allows the macOS device password as a fallback when Touch ID is unavailable (e.g., wet fingers, too many failed attempts). This means "Touch ID" in this document also includes the macOS password as a last resort on the macOS side.
